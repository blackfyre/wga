# Dual Mode Merge Readiness

## Introduction

This branch is now the prospective application baseline because it includes the agreed framework and project-structure upgrades. The remaining delivery work is therefore limited to bringing Dual Mode back to `docs/features/dual-mode.md`: a desktop-only artist/artwork comparison view with shareable pane state, a minimal menu-and-chooser entry flow, and no unrelated promotional or convenience behaviour.

## Phase 1 — Establish the retained baseline

### Item: Separate the baseline upgrade from Dual Mode scope

- [✓] Confirm the working tree is clean and retain the branch’s framework, dependency, and `internal/` structure migration. **Done when:** no framework migration files are reverted merely to reduce the Dual Mode diff.
- [✓] Identify Dual Mode-only files and behaviour to trim. **Done when:** retained and removed behaviour can each be traced to the feature specification or an explicit user decision.

## Phase 2 — Reduce the feature to its agreed behaviour

### Item: Make pane navigation explicit and symmetric

- [✓] Update `internal/handlers/dual/main.go` so each source pane has a validated target pane, defaulting to the other pane. **Done when:** a link updates the source or opposite pane according to its persisted control state, without changing the untouched pane.
- [✓] Preserve target choices in the canonical Dual Mode URL. **Done when:** reload and shared URLs retain both selected content and both target choices.
- [✓] Update `internal/handlers/dual/main_test.go` with table-driven URL and target-state regression coverage. **Done when:** malformed targets fall back safely and both source panes behave symmetrically.

### Item: Keep only the minimum interface

- [✓] Update `internal/assets/templ/pages/dual.templ` to retain the desktop split view, chooser, and two labelled bottom-bar target checkboxes. **Done when:** each checkbox clearly controls whether links from its pane open in the other pane.
- [✓] Remove clear, copy, reverse, and Dual Mode promotional content that is not required by the feature specification. **Done when:** no removed action remains reachable from the Dual Mode page or its default-pane guidance.
- [✓] Update `resources/js/app.ts` to support only chooser actions and target-checkbox changes. **Done when:** the exposed `window.wga.dual` API contains no obsolete control actions and target changes refresh canonical server-rendered content.
- [✓] Regenerate Templ and JavaScript outputs rather than hand-editing generated files. **Done when:** source and generated assets are in sync.

### Item: Meet default-state resilience requirements

- [✓] Make an unsupported or unavailable pane fall back locally without preventing the other pane from rendering. **Done when:** an empty, malformed, or missing pane selection produces a valid default state and the other valid pane remains visible.

## Phase 3 — Prove merge readiness

### Item: Cover the cross-boundary user journey

- [✓] Add a focused `playwright-tests/dual-mode.spec.ts` journey. **Done when:** desktop layout, mobile unsupported message, chooser loading, target-checkbox behaviour, URL persistence, and missing-pane fallback are asserted from the browser boundary.
- [✓] Run template generation, formatting, focused Go tests, full Go tests, the new Playwright test, the complete Playwright suite, and `app:build`. **Done when:** focused checks, build, and the four Dual Mode browser checks pass; the full suite ran with 12/15 passing, while pre-existing Guestbook locator ambiguity, Feedback toast text, and postcard CAPTCHA configuration failures remain outside this delivery.
- [✓] Review `git diff --check`, final status, and the final diff against the retained baseline. **Done when:** changes are limited to Dual Mode source, its generated artefacts, and focused tests.
- [ ] Manual check by the user. **Done when:** the desktop comparison workflow and bottom-bar target controls match the agreed interaction.
- [ ] Commit and push. **Done when:** explicitly requested by the user after review.
